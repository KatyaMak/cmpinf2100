The Execution Stack

Every line of Python code you write travels down this stack before anything actually computes. The question is: how far down does it go, and how many times?
Python
Where you write code. Where map() and pandas .apply() live and loop.
↓
Pandas
Sits on top of NumPy. Adds DataFrames, Series, indexes. .apply() belongs here.
↓
NumPy
Array engine. Vectorized operations start here — they don't loop in Python, they hand off immediately.
↓
Compiled C
Pre-compiled, optimized loops. NumPy passes entire arrays here. No Python overhead.

Key insight: All three approaches — map, apply, vectorized — eventually reach machine code. The difference is how many times they make the trip, and which level does the looping.

map vs apply — Same Pattern, Different Worlds

Map vs Apply — they share a pattern — apply a function to elements — but they belong to completely different layers of the stack and work on different data structures.

Python's map()

Built-in Python
Works on any iterable: lists, tuples, anything Python can loop over.

Applies a function to each element. Returns a lazy iterator — nothing runs until you consume it. 
nums = [1, 2, 3, 4]
result = map(lambda x: x * 2, nums)
list(result)  # → [2, 4, 6, 8]
Loops in Python. Each item: Python → C → machine → back up.

Pandas .apply()

Pandas only
Works on Series (one column) or DataFrame (rows or columns via axis=).

Applies a function across a pandas structure that sits on top of NumPy arrays. 
df['col'].apply(lambda x: x * 2)
# or across rows:
df.apply(lambda row: row['a'] + row['b'],
         axis=1)
Still loops in Python. Same bouncing, same cost.

Different structures, different levels — same kind of Python-level loop underneath.

Wait — there are two things called .map()

Naming collision. Python has a built-in map(). Pandas has its own .map() method on a Series. Same name — completely different things.

Python's built-in map()

Built-in Python
A standalone function. Works on any iterable — lists, tuples, generators. Has nothing to do with pandas. 
nums = [1, 2, 3]
list(map(lambda x: x * 2, nums))
# → [2, 4, 6]
# Pure Python. No DataFrame involved.
You call it as map(function, iterable)

Series.map() — pandas method

Pandas only
A method that lives on a pandas Series. Only works on a single column. Can take a function, a dictionary, or another Series as input. 
df_2["Team_with_acronym"] = df_2.Team.map(append_abbreviation)
# df_2.Team is a Series
# .map() belongs to that Series
# Applies append_abbreviation to each element
You call it as series.map(function) — dot notation, on the column

In your lab: df_2.Team.map(append_abbreviation) — this is pandas .map(), not Python's map(). df_2.Team is a Series, and .map() is a method on that Series. The fact that both are called "map" is genuinely confusing. The key: if you see dot notation on a column — series.map() — it's pandas. If you see it as a standalone function — map(fn, list) — it's Python.

And the third option: vectorized NumPy

Instead of calling a function on each element, you write an operation that acts on the entire array at once. NumPy hands it to compiled C. No Python loop. 
import numpy as np
arr = np.array([1, 2, 3, 4])
result = arr * 2   # → array([2, 4, 6, 8])
# No loop written. NumPy did it in C.

The Trip Cost — Up and Down the Stack

Every time code moves between Python and the compiled layer, there's overhead — the cost of switching context, passing data, returning results. The number of those trips is what separates slow from fast.

map / apply — Many Trips

Vectorized NumPy — One Trip

Same destination, different frequency. Both approaches reach machine code. With map/apply: Python makes the round trip for every single item. With vectorized: Python hands the whole array to C once, C does all the looping internally, returns the result.

Integer and boolean columns are stored as contiguous NumPy arrays — they can be vectorized directly. String columns (dtype=object) store pointers to Python string objects scattered in memory — they can't be vectorized the same way, which is one reason apply or map are often used on string columns.

Why the structure determines the tool

Numeric column → vectorize

Stored as a contiguous NumPy array. You can pass the whole thing to C in one trip. 
df['score'] * 1.1
# NumPy multiplies entire array
# One trip to C
✓ vectorizable

String column → apply or map

Stored as pointers to Python string objects. No contiguous block to hand to C. 
df['name'].apply(str.upper)
# Loops through each pointer
# Python-level iteration
not easily vectorizable

Simple transformation → vectorize

Uniform operation across the array. NumPy knows how to batch it. 
df['score'] > 80
# Returns boolean array
# One C operation
✓ vectorizable

Conditional logic → apply

When each row's result depends on multiple conditions or columns, NumPy can't batch it. 
df.apply(
  lambda r: 'high'
    if r['score'] > 80
    else 'low', axis=1)
needs apply

The pointer detail — strings vs numbers

When pandas stores integers, they sit next to each other in RAM — 88, 74, 91 in a row. C can sweep through them in one pass. When pandas stores strings, each cell holds a pointer (an address) that points to a Python string object somewhere else in memory. C can't sweep through those the same way — it has to follow each pointer separately. That's why string operations stay at the Python level.

select_dtypes() — filtering by what's actually in memory

Because numeric columns and string columns are stored differently, it often makes sense to filter to numeric columns first before running operations. select_dtypes() is how you do that — it returns only the columns whose underlying storage matches what you asked for.

What select_dtypes does

Filters the DataFrame to columns of a given type. The result is a smaller DataFrame — only the matching columns, same rows. 
df.select_dtypes("number")
# Returns only int and float columns
# Drops strings, booleans, etc.

df.select_dtypes("object")
# Returns only string columns

Why it connects to the stack

select_dtypes("number") gives you only contiguous NumPy arrays. That means everything in the result can take one trip to C. You're not just filtering — you're selecting the columns that are actually stored in the right format for fast operations.

When to Use What

In practice you won't stop and think "which layer am I using?" — but understanding the stack tells you why certain choices are faster, and what to reach for when performance matters.

Decision guide

SituationReach forWhy
Math on a numeric column: multiply, add, compare Vectorized NumPy / pandas Contiguous array → one C trip
Transform one column with a simple function .map() on Series Readable, works element-by-element
Operation needs multiple columns in the same row .apply(axis=1) Sees the whole row at once
Complex conditional logic per row .apply() Can't vectorize conditional branching
String cleaning, text operations .apply() or .str. methods Strings stored as pointers, not arrays
Loop over any Python list or iterable Python map() Not in pandas yet, pure Python
Small dataset, any approach Anything readable Speed difference negligible under ~1000 rows
Large dataset, numeric operation Always vectorize 10,000+ rows: trip cost multiplies fast

The mental shortcut

Can NumPy do this in one pass?

Uniform operation on numbers? → vectorize

Conditional, multi-column, or strings? → apply / map

What structure is your data in?

Python list → Python map()

Pandas Series → .map() on Series

Pandas DataFrame → .apply(axis=0/1)

What your class asked you to remember

Your lab gave you two lines of code. Now you know what they're actually doing: 
# map — Python built-in, loops over any iterable at Python level
list(map(lambda x: x * 2, [1, 2, 3]))

# apply — pandas method, loops over Series/DataFrame rows or columns at Python level
df['col'].apply(lambda x: x * 2)

# vectorized — skips Python loop, hands array to C in one trip
df['col'] * 2   # same result, much faster at scale
All three produce the same result. The difference is which layer does the looping, and how many trips down the stack.
You will use vectorized operations constantly in data science — not consciously as "I am vectorizing now," but every time you write df['col'] * 2 or arr > threshold, that's it. apply is for the cases that can't be vectorized. Python's map is for when you're not in pandas yet.
